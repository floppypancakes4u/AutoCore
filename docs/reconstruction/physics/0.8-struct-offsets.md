# Phase 0.8 — Authoritative vehicle physics struct-offset map

Cross-referenced from the six known driver functions in `autoassault.exe` (base `0x400000`,
project **AA-decode**). All offsets are in **bytes**, hex. "Evidence" names the decompiled
function(s) that prove the access.

Source functions:
| Fn | Addr | Role |
|----|------|------|
| `VehicleAction_applyAction`     | `0x598650` | per-tick havok vehicle driver (this = Havok VehicleAction) |
| `VehicleAction_calcWheelTorque` | `0x598040` | per-wheel engine torque → friction solver |
| `VehicleAction_airStabilization`| `0x598320` | AVD damping + collision recovery |
| `PushDriveAxesToController`     | `0x4fbc10` | entity drive axes → input controller (this = CVOGVehicle entity) |
| `MoveToTarget3DPoint`           | `0x4fc650` | AI steer/throttle toward aim point (this = CVOGVehicle entity) |
| `FUN_0053eec0` (net apply)      | `0x53eec0` | network state apply (pos/rot/vel/angvel) |

Entity→data chain (for movement mode): `ent+0x258 → [+4]+4 → +0xac+ent → +0x3c`:
- `+0x4cc` = rear-wheel start index (wheels ≥ this = "rear")
- `+0x4ce` = **movement-mode byte** (`0x02` = analog ramp path)
- `+0x634` = **speed cap / AbsoluteTopSpeed** (f32; `-1` = no cap) — *on vehicleData, reached
  via `[ent+0xac]+0x3c+0x634`, NOT the entity itself.*
- `+0xe4f8` (off `[ent data]+0xa8`... via calcWheelTorque chain) = anim/power gate (nonzero to apply torque)
- `[data]+0xa8 +0x7d/+0x7e` = animation-suspend / fully-disabled flags (applyAction, airStabilization)

## 2. Havok VehicleAction — `this` of applyAction (owned by havok world, back-ref `+0x44`)

| Off | Field | Type | Evidence |
|----:|-------|------|----------|
| `+0x1c` | in-collision flag | u8 | airStabilization sets 1 in window, 0 on recovery |
| `+0x20` | **throttle** (current) | f32 | applyAction ramp `*(this+0x20)*dt`; PushDrive writes throttle here |
| `+0x24` | **steering-ramp stage 1** (ramps `ent+0x618` → wheelsDesc+0x1c) | f32 | applyAction; ⚠ *NOT brake* |
| `+0x28` | **steering final** (stage-1 × min(|v|/0.6,1)) → `setSteeringAngle` | f32 | applyAction mode-0x02 |
| `+0x2c` | **all-wheels-airborne flag** (1 = no wheel in contact) | u8 | calcWheelTorque writes; applyAction gates boost |
| `+0x30` | **boost timer** | f32 | applyAction boost/impulse block |
| `+0x34` | **boost cooldown timer** | f32 | applyAction |
| `+0x3c` | **havok vehicle/steering instance ptr** (`+0x5c` = steering-active byte) | ptr | applyAction `*(this+0x3c)+0x5c=1` |
| `+0x40` | **wheel container** (`+0xc` = hkDefaultWheels desc) | ptr | applyAction, calcWheelTorque |
| `+0x44` | **entity back-ref** (CVOGVehicle) | ptr | every access `this+0x44` |

wheelsDesc = `[VehicleAction+0x40]+0xc`:
- `+0x0c` = wheel count
- `+0x14` = steering-desc ptr (`+0x1c` = steering-angle input written by applyAction)
- `+0x28` = per-wheel **engine-torque output array** (f32[]; calcWheelTorque writes `[+0x28]+i*4`)
- `+0x80` = per-wheel **runtime state array** (stride `0xc0`, see §4)

### RESOLVED — `+0x30..+0x3c` is a QUATERNION, not matrix rows
A 3×3/3×4 matrix from `+0x30` would overlap the linear-velocity vector at `+0x40`; only a
4-float quaternion (`+0x30..+0x3c`) fits before `+0x40`. `FUN_004e8a40 / 004e8b60 / 004e8ad0`
are **quaternion → basis-vector extractors** (right / up / forward): calcWheelTorque passes
`body+0x30` to `004e8b60` and dots the result with world-up (`DAT_00af3390/94/98`) to get the
chassis up-axis for the upright check. applyAction's non-0x02 branch expands the same quaternion
(`q.w²`, cross-products `q.y·q.z−…`) into a rotation to derive desired angular velocity.
`+0xb0` is unambiguously the **world position** vec3 (x,y,height,z).

## 4. WHEEL runtime state — array at `wheelsDesc+0x80`, stride `0xc0`

| Off | Field | Type | Evidence |
|----:|-------|------|----------|
| `+0x20` | torqueCurve2D **arg1** (wheel rpm/speed; from preUpdate contact kinematics) | f32 | calcWheelTorque `torqueCurve2D([+0x20],[+0x28])` |
| `+0x28` | torqueCurve2D **arg2** (throttle proj) | f32 | calcWheelTorque |
| `+0x80` | **in-contact byte** (0 → torque forced 0) | u8 | calcWheelTorque gate; applyAction |
| `+0x88` | **axle factor** (postTick aggregates torque × this ÷ axleWheelCount) | f32 | annotation from `hkVehicleFramework_postTickApplyForces 0x64bc70` |
| `+0x90` | wheel spin angle (`+= dt*spinSpeed`, preUpdate) | f32 | cross-ref `0.1-step-rate.md` / hkVehicleFramework_preUpdate |
| `+0xb0` | **suspension compression length** (negative = compressed; min used to lift chassis) | f32 | applyAction: scans `[+0xb0]`, if `<0` raises body pos by `-min` |

Per-wheel engine torque **output** is written to the separate array `wheelsDesc+0x28[i]` (f32),
clamped `[0, 1000.0]`; it is the drive-impulse input to `hkVehicleFrictionSolver_solve 0x6c4450`.

## 5. Network apply `FUN_0053eec0` — field routing (corroborates the body layout)

Args: `param_2`=position, `param_3`=rotation, `param_4`=linear velocity, `param_5`=angular
velocity, `param_6`=extra. Routes into an **interpolation ghost** `param_1[10]` (created by
`FUN_0053e020`):
- ghost `+0x00..0x0c` = position, `+0x10..0x1c` = rotation (if `FUN_00568100`), `+0x20..0x2c`
  = linear velocity (or `DAT_00b04610..` if below `DAT_00a0f718`), `+0x30..0x3c` = angular velocity.
- When authoritative (owns body) and displacement `> DAT_009d000c`: snaps live body via vtable
  `+0x40` (set transform) + `ApplyImpulseVector(param_4)`; else stores to entity `+0x84`
  (position) / `+0x94` (rotation) fallback slots (used when `physicsObj` is null everywhere).

**Entity fallback (no havok body):** `[[ent+4]+4] + 0x84 + ent` = position vec, `+0x94` = rotation
— confirmed identically in applyAction, calcWheelTorque, MoveToTarget, and net-apply.

## Corrections vs. the incoming assumptions
- **VehicleAction `+0x24` is the steering-ramp stage, NOT brake**; `+0x28` is the speed-scaled
  final steer angle. No dedicated brake float exists on the havok action (braking = negative
  throttle / handbrake byte).
- **Body `+0x30..0x3c` = quaternion** (x,y,z,w), not matrix rows.
- **Body `+0x50..0x5c` = angular velocity** (extends the map beyond the requested fields).
- **Entity `+0x634` speed cap lives on vehicleData** (`[ent+0xac]+0x3c+0x634`), not on the entity.
- **PushDrive's controller (`[ent+0x1a0]+8`) is a distinct input struct**, not the havok
  VehicleAction (its `+0x24` is a handbrake byte, overlapping the action's `+0x24` float).
- Added wheel `+0x90` (spin angle) and body `+0x2c` (inverse mass), `+0x4c/+0xbc` pad/W lanes.
