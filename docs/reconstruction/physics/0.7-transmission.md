# 0.7 — Transmission / Engine-RPM model feeding the torque curve

Binary: `autoassault.exe` (Ghidra project, image base 0x400000). Read-only analysis.

## TL;DR (headline correction)

The task premise — that `torqueCurve2D(wheel+0x20, wheel+0x28)` is indexed by **engine RPM ×
throttle** — does **not** hold. Tracing the writers shows `wheel+0x20` and `wheel+0x28` are the
**X and Z components of the wheel↔ground CONTACT HARDPOINT position vector** (world space), written
by `hkVehicleFramework_preUpdate`. AA replaced Havok's `hkVehicleDefaultEngine`/`Transmission`
runtime with `VehicleEngine::torqueCurve2D`, a 2-D byte-LUT. Because the LUT is binned against
world coordinates (typically far outside `[0, cols]`), it almost always falls through to the
default entry `float[0]` at `engine+0x344` — i.e. the "torque curve" degenerates to a near-constant
torque factor. The classic gear/RPM shift machinery in `VehicleSpecific` is parsed from the DB but
is **not** what shapes per-wheel drive torque through this path.

## What writes wheel+0x20 / wheel+0x28 (the trace)

`hkVehicleFramework_preUpdate` @ 0x64cf20, per-wheel loop 2 (stride 0xC0):

```
pfVar1 = wheel+0x20                       // dst = 4-float vec at +0x20..+0x2c
*pfVar1      = wheel+0x00                  // = hardpoint origin (chassis-space attach pt)
wheel+0x24  = wheel+0x04
wheel+0x28  = wheel+0x08
wheel+0x2c  = wheel+0x0c
len = (suspLimit + suspLen) * fStack_94    // fStack_94 defaults 1.0
wheel+0x20 += wheel+0x50 * len             // + suspension-direction unit vec (wheel+0x50..0x5c)
wheel+0x24 += wheel+0x54 * len
wheel+0x28 += wheel+0x58 * len
wheel+0x2c += wheel+0x5c * len
```

So `wheel+0x20..+0x2c` = **hardpoint (`wheel+0x00`) + suspensionDir (`wheel+0x50`) × suspensionLength**
= the contact/hardpoint world position. `calcWheelTorque` reads `.x`(+0x20) and `.z`(+0x28) of that
vector and passes them to `torqueCurve2D`. There is **no** RPM or throttle writer to these offsets; no
child component or postTick overwrites them between preUpdate and calcWheelTorque.
(Matches prior annotation WI-MOV-003b.)

## Where the actual wheel RPM/spin lives

preUpdate loop 3 (the true kinematics/"engine-RPM" analogue):

```
spin = (wheel+0x9c + chassisContactVel) / wheelRadius[i]   // wheelRadius = wheelData+0x10 array
wheel+0x8c = spin                                          // wheel angular velocity
wheel+0x90 += dt * spin                                    // accumulated spin angle
```

`wheel+0x9c` is the friction/drive feedback written back by postTick the previous tick. This is
Havok's standard `spinVelocity = groundContactSpeed / radius`. Engine RPM (had AA used it) would be
`spin × TransmissionRatio × GearRatio[gear]`. **This value (wheel+0x8c) is never passed to
torqueCurve2D** — confirming the torque curve is not RPM-driven.

## torqueCurve2D internals (0x4a9750)

```
engine+0x0c : enabled byte (0 → return 1.0)
engine+0x18 : bin range scale
engine+0x10 : rows (X bins), engine+0x14 : cols (Y bins)
engine+0x3dc: rows*cols byte table (each &7 = index)
engine+0x344: float[8] factor values; [0] = default/out-of-range

base = engine+0x18 * 0.5;  inv = 1/(engine+0x18)
xbin = int((param2 - base) * inv);  ybin = int((param3 - base) * inv)
if in-range: return float[ table[cols*xbin + ybin] & 7 ]
else:        return float[0]          // <-- dominant path when args are world coords
```

### Up/down-shift logic (Havok standard, from reflected params)
Standard `hkVehicleDefaultTransmission::calcTransmission`: if `engineRPM > upshiftRPM` and
`gear < NumberOfGears-1` → gear++ (after `clutchDelayTime`); if `engineRPM < downshiftRPM` and
`gear > 0` → gear--. `engineRPM = |wheelSpin| × primaryTransmissionRatio × gearRatio[gear]`,
clamped to `[minRPM, maximumRPMMax]`. **AA runtime does not exercise this against torqueCurve2D**
(see postTick note: "NO read of transmission +0x1c; AA has no hkDefaultEngine; transm torque is
residual"). Whether a transmission child still steps gear state for HUD (`i_d_speed_2d_gauge_rpm.xml`,
0xa512bc) is unconfirmed.

## calcWheelTorque per-wheel formula (0x598040)

```
if !inContact(wheel+0x80): wheels+0x28[i] = 0
else:
  t = torqueCurve2D(wheel+0x20 /*contact.x*/, wheel+0x28 /*contact.z*/)   // ≈ float[0] const
  driverMod (entity vtbl+0x214 → +0x118, default 0):
     >0: t = 1-(1-mod)(1-t)
     <0: rear wheels ×2.0; t *= (1+mod)
  upright = 1.0, else pow-falloff if |dot(bodyUp,worldUp)| < 0.8
  μ = wheelsetFriction[i] (FUN_004f5550)
  if |v| < 15.0: μ *= (15-|v|)*0.2 + 1     // low-speed traction boost
  torque = μ * upright * t
  if (entity+0x61c && rear): torque *= 0.5  // handbrake/burnout rear cut (NOT RearWheelFrictionScalar)
  clamp [0, 1000.0] → wheels+0x28[i]
```

## Ambiguities / open items

1. **Biggest**: the "engine torque curve" is indexed by contact-position X/Z, not RPM/throttle. Either
   (a) AA intentionally repurposed the 2-D LUT as a near-constant torque factor (float[0] fallthrough),
   or (b) the LUT is a terrain/position-dependent traction map. `engine+0x18` range and the `float[8]`
   table contents are needed at runtime to decide — not resolvable statically here.
2. Whether any of the 7 subsystem children (tickSubsystems `param_1[5..0xb]`) steps gear/RPM state for
   HUD only vs. physics is unconfirmed.
3. VehicleSpecific numeric offsets for each transmission field: `VehicleDb_LoadCloneBase` reads columns
   in SQL order into locals then scatter-copies to the struct; a clean column→offset table was not
   extracted (loader is a 200+ column sequential reader). Reflection-name↔column mapping above is solid;
   exact struct byte offsets are TODO.
