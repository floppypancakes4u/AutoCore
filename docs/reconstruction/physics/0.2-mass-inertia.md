# Phase 0.2 — Vehicle Chassis Mass, Inertia Tensor & Center of Mass

Reverse-engineered from `autoassault.exe` (Ghidra project `AA-decode`, image base 0x400000,
Havok 2.3 SDK). Read-only investigation — no DB edits.

## TL;DR (for the server port)

| Quantity | Source in client | Server has it? |
|---|---|---|
| **Chassis rigid-body mass (kg)** | **Baked into the Havok physics asset (`.glm`) named by `tSimpleObject.strPhysicsName`.** Loaded by the rigid-body loader `FUN_004f1180`; the mass lives inside the serialized `hkpRigidBody`. NOT a clonebase scalar. | **No** (asset-only). Must be approximated. |
| **Chassis inertia tensor (diagonal)** | Havok raycast-vehicle uses **unit (per-unit-mass) inertia** `chassisUnitInertia{Yaw,Roll,Pitch}` in `hkVehicleData`. These are filled from the vehicle definition struct at **+0x5e4 / +0x5dc / +0x5e0**, which mirror the DB columns **`rlRVInertiaYaw/Roll/Pitch`**. Actual inertia = `mass * unitInertia`. | **Yes** — `RVInertiaRoll/Pitch/Yaw`. |
| **Center of mass** | Baked into the physics asset's `hkpRigidBody`; the DB `fCenterOfMassModifierX/Y/Z` (`tVehicle`) is a delta applied on top. | **Modifier: yes.** Base COM: asset-only. |
| **Friction equalizer** | `hkVehicleData.frictionEqualizer` ← DB `rlRVFrictionEqualizer`. | Yes. |

**Bottom line:** the only genuinely missing scalar is the **absolute chassis mass**. Everything the
vehicle *simulation* consumes (`hkVehicleData`) is normalized *per unit mass*, so the server can pick
any consistent mass and get correct handling as long as engine torque / impulse magnitudes are scaled
to match. See "Recommended approximation" below.

---

## 1. Where the physics object (chassis rigid body) is built

`Vehicle_createVehicleAction` (**0x4fb6a0**) → `Vehicle_buildHavokVehicleFramework` (**0x5fd390**),
the sole constructor of every `hkVehicle*` component. The chassis rigid body itself is NOT built
here — it is the entity's main physics collision body, created earlier by the generic clonebase
rigid-body loader:

- **`FUN_004f1180`** (clonebase rigid-body loader, from `CloneBase.cpp`; asserts
  `"!!!!!Rigid body of object '%s' failed to load: %s."` and
  `"REAL BAD. A PHYSICS OBJECT FAILED TO LOAD PROPERLY %s"` @ 0x9ccf68).
  - Reads the **physics asset name** from `clonebase+0x40a` (`strPhysicsName`, a wchar string;
    `clonebase+0x3c` → definition, `+0x205` wchars in).
  - Calls `FUN_004ee080(...)` which loads/instantiates the serialized `hkpRigidBody` and stores it
    at `object+0x98`.
  - For **vehicles (object type `0x12` = 18)** it passes a physics **scale** = `definition+0x4d4`
    (default 1.0) into the rigid-body factory. This scales the asset, and with it the mass/inertia.
  - **No `rlMass` is applied to the rigid body here.** The mass is whatever the `.glm`/`hkx` asset
    serialized.

Physics assets live under `..\physics\`, `..\physics\physx\`, `..\physics\planes\`
(strings @ 0xa95ff8–0xa96084). The Havok class-reflection strings — `chassisMass` (0x9e7344),
`inertiaTensor` (0x9dc6c8), `centerOfMass` (0x9dc6b8), `hkDefaultChassis` (0x9e4ffc),
`hkRigidBody` (0x9dc834) — have **no direct code xrefs**; they are serialization type descriptors,
confirming the rigid body (mass + inertia + COM) is **data-driven from the asset**, not computed at
runtime from clonebase scalars.

## 2. The vehicle-simulation data (`hkVehicleData`) — where inertia actually enters the sim

`Vehicle_buildHavokVehicleFramework` (0x5fd390) builds the framework. Component ctors:

- `hkDefaultWheels_ctor` 0x64fee0, **`hkDefaultChassis_ctor` 0x64fdf0** (wraps the hkRigidBody at
  chassis+0x3c), `hkDefaultSteering_ctor` 0x64fac0 / `TankSteering_ctor` 0x64fc80 (if
  `definition+0x4c0 == 4`), `hkDefaultTransmission_ctor` 0x64f610, `hkDefaultBrake_ctor` 0x64ed40,
  `hkDefaultSuspension_ctor` 0x64e510, `hkDefaultAerodynamics_ctor` 0x64da90,
  `hkAngularVelocityDamper_ctor` 0x64d900, `hkVehicleFramework_ctor` 0x64cd30.
- **No `hkDefaultEngine`** — AA replaced it; engine torque is `VehicleAction_calcWheelTorque`
  (0x598040) + `VehicleEngine_torqueCurve2D` (0x4a9750).

The `hkVehicleData` descriptor is initialized with defaults by **`FUN_00650020`** then populated from
the vehicle definition struct by **`FUN_005fc620`**. The definition struct is reached via
`(entityData + [entityData+4]+4 + 0xac) + 0x3c`.

### `FUN_005fc620` — definition → `hkVehicleData` descriptor mapping

```
desc+0x28 = def+0x5c4        desc+0x38 = def+0x5d8
desc+0x2c = def+0x5c8        desc+0x3c = 10.0   (DAT_00af4618)      [default DAT_00a110d8 = 10.0]
desc+0x30 = def+0x5cc        desc+0x40 = 0.2    (DAT_00af4614)
desc+0x34 = def+0x5d0        desc+0x44 = def+0x5e4   <-- chassisUnitInertia (Yaw)   [default 1.0]
                             desc+0x48 = def+0x5dc   <-- chassisUnitInertia (Roll)  [default 1.0]
                             desc+0x4c = def+0x5e0   <-- chassisUnitInertia (Pitch) [default 1.0]
```

`FUN_00650020` defaults (before override): `desc+0x34=1.0, +0x3c=10.0, +0x40=0.0,
+0x44=+0x48=+0x4c=1.0, +0x50=+0x54=0, +0x58=0x80000000`. The three consecutive slots
`+0x44/+0x48/+0x4c` defaulting to **1.0** are the Havok `hkVehicleData` **`m_chassisUnitInertiaYaw /
Roll / Pitch`** — and they are overridden from **def+0x5e4 / +0x5dc / +0x5e0**.

These definition offsets are the runtime home of the DB columns **`rlRVInertiaYaw` (0xa92fa8),
`rlRVInertiaPitch` (0xa92fc8), `rlRVInertiaRoll` (0xa92fec)** (loaded by
`VehicleDb_LoadCloneBase` @ 0x7efb40; the store site is around 0x7f3211–0x7f32a6). The definition
struct is populated via a bulk copy (no discrete `MOVSS`/`FLD` to +0x5dc–+0x5e4 exist in the setup
path), which is why a byte-pattern search for direct loads returns nothing.

**Havok raycast-vehicle inertia semantics:** `m_chassisUnitInertia*` are per-unit-mass. Havok builds
the chassis rigid body's actual inertia tensor as a **diagonal**, `I_axis = mass * unitInertia_axis`.
The axis→field pairing is now **live-verified** (see §2.1) — it is keyed to the chassis basis
**front = +Z, up = +Y, lateral = ±X**, so each named component sits on the axis it rotates about:

```
InvInertiaZ (about forward/+Z) = 1 / (mass * RVInertiaRoll)   ← Roll
InvInertiaX (about lateral/±X) = 1 / (mass * RVInertiaPitch)  ← Pitch
InvInertiaY (about up/+Y)      = 1 / (mass * RVInertiaYaw)    ← Yaw
```

### 2.1 Live confirmation (Task B4 — Cheat Engine capture, 2026-07-16)

Captured against the single live vehicle in `autoassault.exe` (PID 48600) via the CE bridge
(`tmp/re/ce_client.py` direct pipe). Attach + 1:1 static map re-verified (`0x9cc798`=`41EFFFFF`,
`0xaf3388`=`41A00000`). Breakpoint at `applyAction 0x598650` → `this`(ECX)=`0x3626B580`; single
instance ticking.

**Walk to the live chassis rigid body** (`fn_offsets_rigidbody.md` path B):
`entity = *(this+0x44)` = `0x3d06f250` → `physicsObj = *(entity+0x08)` = `0x3627c3e0` →
`rb = *(physicsObj+0x3c)` = `0x3b0c3940`.

**Body-frame principal inverse inertia + inverse mass at `rb+0xe0`** (Havok
`hkpMotion::m_inertiaAndMassInv`, xyzw = invIx, invIy, invIz, invMass) — stable across reads and
across a non-identity orientation (quat ≈ `[0.018, 0.428, 0.030, −0.903]`), confirming it is the
**body/principal frame**, not world:

```
rb+0xe0 = (0.00022222, 0.00022222, 0.00066667, 0.00066667)
        = (1/4500,     1/4500,     1/1500,     1/1500)
→ body inertia diag  I = (Ix=4500, Iy=4500, Iz=1500),  mass = 1/invMass = 1500
→ unit inertia (I/mass) = (X=3, Y=3, Z=1)
```

**Chassis basis, live** (`vehicleData = *(fw+0x10)` = `0x36599640`, `fw = *(this+0x40)`):
`+0x10` front axis = `(0,0,1)` (**+Z**), `+0x20` up axis = `(0,1,0)` (**+Y**),
`+0x30` side axis = `(−1,0,0)` (**−X**), `+0x00` mass = `1500`.

**DB definition RVInertia, live** (`def = *( *(base+0xAC) )+0x3c` where
`base = entity + *(*(entity+4)+4)`; def = `0x2c97e6e0`):

```
def+0x5dc RVInertiaRoll  = 1
def+0x5e0 RVInertiaPitch = 3
def+0x5e4 RVInertiaYaw   = 3
```

**Cross-check (closes the pairing with no free parameters):** `mass * RVInertia` =
`1500 * (Roll=1, Pitch=3, Yaw=3)` = `(1500, 4500, 4500)`. Matching to the measured body-frame
diagonal `(Ix=4500, Iy=4500, Iz=1500)`:

| Body axis | Measured I | = mass × | Named component | Rotation axis |
|-----------|-----------:|----------|-----------------|---------------|
| **X** (lateral) | 4500 | Pitch (3) | **RVInertiaPitch** | about lateral |
| **Y** (up)      | 4500 | Yaw (3)   | **RVInertiaYaw**   | about vertical |
| **Z** (forward) | 1500 | Roll (1)  | **RVInertiaRoll**  | about longitudinal |

This is unambiguous because `RVInertiaRoll=1` is the **only** distinct value and lands on the axis
with the lowest inertia (`Iz=1500`, the forward/+Z axis) — the physical roll axis. It also proves
the asset-baked tensor **is** `mass × RVInertia` (not an independent asset value).

**Port impact (fixed in C1):** the previous `VehiclePhysicsInstance.CreateBody` guessed
`InvInertiaX = 1/InertiaRoll`, `InvInertiaZ = 1/InertiaPitch` — i.e. **Roll and Pitch swapped**
(only `Y = Yaw` was right). Corrected to `X←Pitch, Y←Yaw, Z←Roll`.

**COM:** the base COM is asset-side (unavailable server-side); the port applies
`CenterOfMassModifier` as a lever-arm delta in `HkVehicleData.ComputeLeverArm`
(`r = hardpoint − CenterOfMassModifier`, see `fn_com_modifier.md`), so `CreateBody` needs no COM
term. `rb+0x60` (world COM ≈ `1514.2, 124.7, 1805.8`) vs `rb+0xb0` (origin ≈ `1514, 123.6, 1805.9`)
shows a small live COM offset but its body-local decomposition was not separated this pass
(low priority — mass-normalized handling is unaffected).

`desc+0x28..+0x38` (def `+0x5c4/5c8/5cc/5d0/5d8`) are the remaining `hkVehicleData` scalars
(torque roll/pitch/yaw factors, extra-torque, max-velocity-for-positional-friction). Constants:
`DAT_00af4618 = 10.0`, `DAT_00af4614 = 0.2`, `DAT_00a110d8 = 10.0`.

## 3. Center of mass

- **Base COM**: serialized in the chassis `hkpRigidBody` inside the physics asset (`strPhysicsName`).
  Reflection strings `centerOfMass` (0x9dc6b8) / `CentreOfMass` (0x9e5d6c) are asset descriptors.
- **`fCenterOfMassModifierX/Y/Z`** (`tVehicle`, confirmed by the SELECT @ 0xa93d40 and the UPDATE
  @ 0xa87e90): a Vector3 **delta** applied on top of the asset COM. It is carried in the definition
  struct alongside `fSkirtExtents*` / `fPushBottomUp`. The exact apply site is the physics/chassis
  init (COM offset on the rigid body); the modifier is a straightforward additive translation of the
  local center of mass. Server should apply it as `COM = assetCOM + CenterOfMassModifier`.

## 4. `rlMass` — what it is and is NOT

`tSimpleObject.rlMass` (column @ 0xa946e8) is loaded by `FUN_007e59d0` and stored at
**object/definition +0x2c** (store at 0x7e6c9c). It is the object's generic "mass" used for
inventory/loot ("Weight:" UI, `rlMassPercent` prefix scaling) and gameplay — **it is not written to
the chassis rigid body** in `FUN_004f1180`. Treat it as a plausible *nominal* mass number but not as
the authoritative Havok chassis mass.

## 5. Recommended server-side approximation (mass truly not in server data)

The absolute chassis mass is asset-only, so the server cannot reproduce it exactly. Two viable ports:

**Option A — Unit-mass sim (recommended, highest fidelity to what the client sim consumes).**
Set `chassisMass = 1.0` and use the inertia tensor **directly equal to `RVInertia{Roll,Pitch,Yaw}`**
(since `I = mass * unitInertia` and mass = 1). Because `hkVehicleData` is entirely normalized per
unit mass (torque factors, friction equalizer, suspension are all unit-relative), the vehicle's
*handling* (steering response, roll/pitch/yaw dynamics, traction) matches the client. Then scale the
AA engine torque (`calcWheelTorque`) and any external impulses by the same nominal factor so
accelerations look right. This avoids inventing a fake kilogram value.

**Option B — Footprint-box mass, if you need believable collision/impulse response vs. other bodies.**
Estimate `mass = density * boxVolume`, `boxVolume = 8 * SkirtExtentsX * SkirtExtentsY * SkirtExtentsZ`
(SkirtExtents are half-extents; from `tVehicle`), with a tuned `density` (e.g. ~150–400 kg/m³ to hit
plausible 1000–3000 kg for AA-scale cars). Then
`I_axis = mass * RVInertia_axis`, and `COM = boxCenter + CenterOfMassModifier`. Use this when the
server arbitrates vehicle-vs-vehicle or vehicle-vs-world impulses and needs comparable momentum.

Either way the **inertia ratios and all handling come from `RVInertia*` + `RVFrictionEqualizer` +
the `tVehicle` tuning columns, which the server already has** — only the single mass scalar is a free
parameter.
