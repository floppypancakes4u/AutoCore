# 0.6 — Havok Vehicle Aerodynamics (drag + lift/downforce + extra-gravity)

Program: `autoassault.exe` (Ghidra project **AA-decode**, image base 0x400000). Havok 2.3 vehicle SDK.
Read-only RE. This is AA's `hkDefaultAerodynamics` component (a standard Havok
`hkVehicleAerodynamics` subclass), ticked once per physics step.

## Functions

| Addr | Name | Role |
|------|------|------|
| `0x0064dae0` | `hkDefaultAerodynamics_update` | Computes & accumulates the aero force. **This is the force application.** |
| `0x0064da90` | `hkDefaultAerodynamics_ctor` | Copies the 8-word descriptor into the runtime component at +0x30..+0x4c. |
| `0x005fc4f0` | `Vehicle_BuildAerodynamicsDescriptor` | Reads VehicleSpecific fields → descriptor. |

| `0x00598650` | `VehicleAction_applyAction` | Per-tick driver; calls tickSubsystems. Does **not** compute aero itself. |

Vtable `PTR_FUN_0x009e4b20`: slot **+0x14 = `0x0064dae0`** (the tick), slot +0x18 → `"airDensity"`
string `0x009e4bdc`. Confirms `hkDefaultAerodynamics_update` is the aerodynamics subsystem's
per-tick method invoked by `VehicleAction_tickSubsystems`.

| desc[i] | struct offset | field |
|---------|--------------|-------|

| desc[1] | +0x59c | **AerodynamicsFrontalArea** |

| desc[3] | +0x5a4 | **AerodynamicsLift** (coefficient) |

Clean sequential struct layout: FrontalArea(0x59c), Drag(0x5a0), Lift(0x5a4), AirDensity(0x5a8),
ExtraGravity[3](0x5ac..0x5b4). Reflection field names confirmed via strings
`rlAerodynamicsFrontalArea/Drag/Lift/AirDensity/ExtraGravityX/Y/Z` (`0x00a932c8` down to `0x00a9319c`).

`_ctor` copies descriptor[0..7] → component `param_1[0xc..0x13]` (byte offsets **+0x30..+0x4c**):

| component offset | value |
|------------------|-------|
| +0x30 | airDensity (rho) |
| +0x34 | frontalArea (A) |
| +0x38 | dragCoefficient (Cd) |
| +0x3c | liftCoefficient (Cl) |
| +0x40 / +0x44 / +0x48 / +0x4c | extraGravity x / y / z / w |
| +0x10..+0x1c | **output force vector** (written by update, read by force applicator) |

## Runtime inputs inside `update`

- `param_1+8` = framework/context. `*(param_1+8)+0x10` = `hkVehicleData` (`iVar15`); `*(param_1+8)+0x30`
  → `+0x3c` = chassis rigid body (`iVar17`).
- `hkVehicleData` axes (chassis-local unit vectors): **frontAxis** = `iVar15+0x10/0x14/0x18`,
  **upAxis** = `iVar15+0x20/0x24/0x28`.
- Chassis body: linear velocity = `body+0x40/0x44/0x48`; world rotation matrix R at `body+0x80`
  (cols/rows at +0x80, +0x90, +0xa0). `body+0x2c` = **inverse mass** (see ExtraGravity).

`worldFront = R * frontAxis`; `worldUp = R * upAxis`.
`v = dot(linearVelocity, worldFront)` = signed **forward speed** (`fVar21`).

## Force formulas

Constants: `DAT_00a0f298 = 0x3f000000 = +0.5`; `DAT_00aaa6cc = 0xbf000000 = -0.5`.

**Drag** (magnitude `dragMag`, applied along worldFront):
```
dragMag = -0.5 * airDensity * frontalArea * dragCoeff * |v| * v      // DAT_00aaa6cc = -0.5
F_drag  = dragMag * worldFront
        = -0.5 * rho * A * Cd * |v|*v  along +frontAxis(world)
```
Because of the `|v|*v` term the drag always **opposes the forward velocity component** (points
backward when moving forward, forward when reversing). Note: drag uses only the *forward* speed
component, not the full velocity magnitude.

**Lift / downforce** (magnitude `liftMag`, applied along worldUp):
```
liftMag = 0.5 * airDensity * frontalArea * liftCoeff * v^2           // DAT_00a0f298 = +0.5
F_lift  = liftMag * worldUp
        = +0.5 * rho * A * Cl * v^2  along +upAxis(world)
```
**Lift sign: along +upAxis** (chassis up, world-transformed). A **positive `liftCoeff` pushes the
car UP** (lift); a **negative `liftCoeff` gives downforce**. `v^2` is always positive so direction is
set purely by the sign of `liftCoeff` (and the up axis).

**ExtraGravity** (added last, world-space, NOT rotated by chassis):
```
invMass = (body+0x2c)                 // inverse mass
mass    = 1/invMass  (0 if invMass==0)
F_out += extraGravity * mass          // per component x,y,z,w
```
ExtraGravity is treated as an **acceleration** (like gravity, world-space constant vector); it is
converted to a force by multiplying by mass and added directly, so the accumulated `+0x10` vector is
a consistent force. It is applied as-authored (a downward pull needs a negative component on the
world "down" axis). No chassis-orientation transform is applied to it.

**Total output** written to `param_1+0x10..0x1c`:
```
F_out = F_drag(worldFront) + F_lift(worldUp) + mass * extraGravity
```
This force vector is consumed by the framework's force applicator after the subsystem tick.

## Ambiguities / notes
- `body+0x2c` is inferred to be **inverse mass** (division then multiply by extraGravity yields a
  force consistent with drag/lift force units). Not 100% confirmed against Havok rigid-body layout.
- frontAxis (+0x10) vs upAxis (+0x20) assignment in `hkVehicleData` is inferred from usage (speed is
  measured along +0x10 → the front axis; lift is applied along +0x20 → the up axis). Consistent with
  Havok defaults but not independently label-verified.
- Drag uses forward-speed `v` (projection on front axis), not full |velocity|. So pure lateral
  sliding produces little aero drag in this model.
- desc[7] / component +0x4c (extraGravity.w) is uninitialized in the descriptor builder and multiplied
  by an all-zero output lane in drag/lift; harmless padding.
