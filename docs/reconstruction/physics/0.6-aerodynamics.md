# 0.6 — Havok Vehicle Aerodynamics (drag + lift/downforce + extra-gravity)

Program: `autoassault.exe` (Ghidra project **AA-decode**, image base 0x400000). Havok 2.3 vehicle SDK.
Read-only RE. This is AA's `hkDefaultAerodynamics` component (a standard Havok
`hkVehicleAerodynamics` subclass), ticked once per physics step.

## Functions

| Addr | Name | Role |
|------|------|------|
| `0x0064dae0` | `hkDefaultAerodynamics_update` | Computes & accumulates the aero force. **This is the force application.** |
| `0x0064da90` | `hkDefaultAerodynamics_ctor` | Copies the 8-word descriptor into the runtime component at +0x30..+0x4c. |
| `0x005fc4f0` | `Vehicle_BuildAerodynamicsDescriptor` | Reads VehicleSpecific fields → descriptor. |
| `0x00636a60` | `VehicleAction_tickSubsystems` | Dispatches vtbl+0x14 (`update`) on each of 7 framework children. |
| `0x00598650` | `VehicleAction_applyAction` | Per-tick driver; calls tickSubsystems. Does **not** compute aero itself. |

Vtable `PTR_FUN_0x009e4b20`: slot **+0x14 = `0x0064dae0`** (the tick), slot +0x18 → `"airDensity"`
string `0x009e4bdc`. Confirms `hkDefaultAerodynamics_update` is the aerodynamics subsystem's
per-tick method invoked by `VehicleAction_tickSubsystems`.

## Data plumbing

`Vehicle_BuildAerodynamicsDescriptor` reads the VehicleSpecific/vehicle-data struct

| desc[i] | struct offset | field |
|---------|--------------|-------|

Clean sequential struct layout: FrontalArea(0x59c), Drag(0x5a0), Lift(0x5a4), AirDensity(0x5a8),
ExtraGravity[3](0x5ac..0x5b4). Reflection field names confirmed via strings
`rlAerodynamicsFrontalArea/Drag/Lift/AirDensity/ExtraGravityX/Y/Z` (`0x00a932c8` down to `0x00a9319c`).

`_ctor` copies descriptor[0..7] → component `param_1[0xc..0x13]` (byte offsets **+0x30..+0x4c**):

| component offset | value |
|------------------|-------|
| +0x30 | airDensity (rho) |
| +0x34 | frontalArea (A) |
| +0x38 | dragCoefficient (Cd) |
| +0x3c | liftCoefficient (Cl) |
| +0x40 / +0x44 / +0x48 / +0x4c | extraGravity x / y / z / w |
| +0x10..+0x1c | **output force vector** (written by update, read by force applicator) |

## Runtime inputs inside `update`

- `param_1+8` = framework/context. `*(param_1+8)+0x10` = `hkVehicleData` (`iVar15`); `*(param_1+8)+0x30`
  → `+0x3c` = chassis rigid body (`iVar17`).
- `hkVehicleData` axes (chassis-local unit vectors): **frontAxis** = `iVar15+0x10/0x14/0x18`,
  **upAxis** = `iVar15+0x20/0x24/0x28`.
- Chassis body: linear velocity = `body+0x40/0x44/0x48`; world rotation matrix R at `body+0x80`
  (cols/rows at +0x80, +0x90, +0xa0). `body+0x2c` = **inverse mass** (see ExtraGravity).

`worldFront = R * frontAxis`; `worldUp = R * upAxis`.
`v = dot(linearVelocity, worldFront)` = signed **forward speed** (`fVar21`).

## Force formulas

Constants: `DAT_00a0f298 = 0x3f000000 = +0.5`; `DAT_00aaa6cc = 0xbf000000 = -0.5`.

**Drag** (magnitude `dragMag`, applied along worldFront):
```
dragMag = -0.5 * airDensity * frontalArea * dragCoeff * |v| * v      // DAT_00aaa6cc = -0.5
F_drag  = dragMag * worldFront
        = -0.5 * rho * A * Cd * |v|*v  along +frontAxis(world)
```
Because of the `|v|*v` term the drag always **opposes the forward velocity component** (points
backward when moving forward, forward when reversing). Note: drag uses only the *forward* speed
component, not the full velocity magnitude.

**Lift / downforce** (magnitude `liftMag`, applied along worldUp):
```
liftMag = 0.5 * airDensity * frontalArea * liftCoeff * v^2           // DAT_00a0f298 = +0.5
F_lift  = liftMag * worldUp
        = +0.5 * rho * A * Cl * v^2  along +upAxis(world)
```
**Lift sign: along +upAxis** (chassis up, world-transformed). A **positive `liftCoeff` pushes the
car UP** (lift); a **negative `liftCoeff` gives downforce**. `v^2` is always positive so direction is
set purely by the sign of `liftCoeff` (and the up axis).

**ExtraGravity** (added last, world-space, NOT rotated by chassis):
```
invMass = (body+0x2c)                 // inverse mass
mass    = 1/invMass  (0 if invMass==0)
F_out += extraGravity * mass          // per component x,y,z,w
```
ExtraGravity is treated as an **acceleration** (like gravity, world-space constant vector); it is
converted to a force by multiplying by mass and added directly, so the accumulated `+0x10` vector is
a consistent force. It is applied as-authored (a downward pull needs a negative component on the
world "down" axis). No chassis-orientation transform is applied to it.

**Total output** written to `param_1+0x10..0x1c`:
```
F_out = F_drag(worldFront) + F_lift(worldUp) + mass * extraGravity
```
This force vector is consumed by the framework's force applicator after the subsystem tick.

## Ambiguities / notes — Task B6 resolution (emulation, 2026-07)

Task B6 built a fully-linked fake struct graph (component → framework →
{vehicleData, holder → chassis body}) and ran `emulate_function` on `0x64dae0`
(Ghidra MCP) against it, in addition to a fresh `decompile_function` +
`disassemble_function` read this session. Golden vectors + fixture:
`src/AutoCore.Game.Tests/Physics/oracles/aero_goldens.json`,
`src/AutoCore.Game.Tests/Physics/oracles/AeroOracleTests.cs`. Both status items
below are **resolved**, one via direct register-level emulation, one via
disassembly (emulation could not directly surface it — see limitation note).

- **`body+0x2c` = inverse mass — CONFIRMED by emulation.** With the chassis body
  struct seeded at `body+0x2c`, the emulator's `XMM1` register at natural
  function return held exactly `1/invMass` (IEEE-754 bit-exact) for three
  different seed values: `invMass=0.1 → XMM1=10.0` (`0x41200000`),
  `invMass=0.25 → XMM1=4.0` (`0x40800000`), `invMass=1.0 → XMM1=1.0`
  (`0x3f800000`). This directly exercises the `UCOMISS`/`JNP`/`DIVSS` sequence
  at `0x64dd00-0x64dd16` in the real binary — no longer purely inferred.
- **frontAxis (`hkVehicleData+0x10`) vs upAxis (`+0x20`) assignment — CONFIRMED,
  but via disassembly, not emulation register readback (tool limitation, see
  below).** Fresh `disassemble_function` of `0x64dae0` shows `EDX+0x10/0x14/0x18`
  (loaded as `iVar15+0x10/14/18`, i.e. `hkVehicleData+0x10`) feed **both** the
  world-front axis used for the drag-direction vector **and** the forward-speed
  scalar `v = dot(linVel, worldFront)` that both `dragMag` and `liftMag` are
  built from. `EDX+0x20/0x24/0x28` (`hkVehicleData+0x20`) feed **only** the
  world-up axis used for the lift-*direction* vector — never the speed scalar.
  This is unambiguous straight-line reading of the instruction stream (no
  branches on this path), independently re-derived this task (not just
  trusting the prior pass's notes).
- **Emulation limitation (why axis assignment isn't emulation-numeric):**
  `hkDefaultAerodynamics::update` returns `void` — its entire result (the
  force xyz/w) is written to memory at `component+0x10..0x1c`, and every
  intermediate (`worldFront`, `worldUp`, `v`, `dragMag`, `liftMag`) is
  recomputed through the **same** XMM0 register in sequence (X, then Y, then Z,
  then W), each overwriting the last before `RET`. The Ghidra MCP
  `emulate_function` tool only surfaces final **register** state, not a
  post-run memory read-back (the scratch addresses used for seeding aren't
  part of the program's real memory map, so `read_memory` fails afterward:
  `Unable to read bytes at ram:...`). Its `max_steps` parameter also does not
  truncate mid-function execution for this straight-line function — a
  `max_steps=1` call still ran to completion and returned deep-in-function
  register values, so bisecting a stop-point wasn't viable either. Net effect:
  only the **last-computed** memory-resident float (`Fw`, always `0.0` here
  since `extraGravity.w=0` in every descriptor we've seen) and any
  register that survives untouched to `RET` (here, `mass` in `XMM1`) are
  emulation-verifiable. `Fx/Fy/Fz` bit-exactness is therefore established via
  hand-derived-from-decompile formulas (same accepted methodology as
  `engine-torque-spec.md` §2 / `fn_004a9750_torqueCurve2D.md`), cross-checked
  against the live `HkVehicleAerodynamics.ComputeForce` port
  (`AeroOracleTests` — 8/8 pass, bit-exact within float tolerance). A live
  debugger (out of scope for this static/emulation-only task) could recover
  `component+0x10..0x1c` directly by breaking at the `RET` and reading process
  memory; that remains open for a future debugger-enabled task if independent
  numeric re-verification of `Fx/Fy/Fz` via a running process is desired.
- Additionally, the emulator completed (`hit_return: true`, no fault) for
  **all 8** seeded scenarios, including two using a non-identity chassis
  rotation matrix — confirming the full pointer chain
  (`param_1+8`→framework, `framework+0x10`→`hkVehicleData`,
  `framework+0x30`→holder, `holder+0x3c`→chassis body) and all documented
  struct offsets are mutually consistent and dereferenceable exactly as this
  doc describes; a wrong offset assumption would very likely have faulted the
  emulator instead.
- Drag uses forward-speed `v` (projection on front axis), not full |velocity|. So pure lateral
  sliding produces little aero drag in this model.
- desc[7] / component +0x4c (extraGravity.w) is uninitialized in the descriptor builder and multiplied
  by an all-zero output lane in drag/lift; harmless padding.
